NLP Semantic Processing Stack

Domain detection · Co-reference resolution · Plotline threading.
Each node: what it is, what it does, where it fits, what else could sit there, and how it hands off to the next stage.

Stage 00 · Input
Raw Text
str / List[str]
Entry point. Unstructured natural language — paragraphs, short stories, agent outputs, documents.
deterministic
Description
Plain unicode string. No assumptions about structure. Could be a sentence, a paragraph, or a full document. The entire downstream pipeline is conditioned on what arrives here.
Where it fits
Source. Everything downstream is a transformation of this.
Alternatives
PDF/DOCX — needs extraction layer first (pdfplumber, docx2txt). HTML — needs stripping (BeautifulSoup). Streaming tokens — requires buffering to sentence boundary before passing forward.
raw string passed to spaCy's nlp(text) — produces a Doc object with token-level annotations
▼
Stage 01 · Foundation NLP
spaCy
en_core_web_trf · v3.x
Tokenization, sentence segmentation, POS tagging, dependency parsing, and named entity recognition in a single pass.
PyTorch · transformer backbone
Description
Industrial-grade NLP library. The trf model uses a RoBERTa backbone for NER and parsing. Produces a Doc object — every token has span, POS, dep-head, and entity label attached. This is the shared data structure that all downstream components read from.
Purpose
Entity extraction (PERSON, GPE, ORG), sentence boundary detection, syntactic structure. The PERSON spans become the character roster for thread tracking.
Where it fits
First real processing stage. Every other component depends on its output — either directly reading the Doc spans, or consuming the sentence list it produces.
Alternatives
flair — higher NER accuracy on literary/fiction text specifically; slower, no dep-parse. stanza — Stanford NLP, strong multilingual support. NLTK — older, no transformer backbone, lower accuracy, fine for toy use. CoreNLP — Java-based Stanford tool; excellent but heavy.
Doc object (token spans + PERSON entities) → FastCoref reads span offsets directly from the spaCy Doc
▼
Stage 02 · Co-reference
FastCoref
LingMess / FCoref · 2023
Resolves pronoun and noun-phrase chains back to their canonical referent. Tells you that "she", "the detective", and "Elena" are the same entity.
PyTorch · span-based
Description
Span-based co-reference model. Returns clusters: each cluster is a list of text spans that refer to the same entity. Two models available — FCoref (fast, less accurate) and LingMess (slow, state-of-the-art). For plotline tracking, LingMess is worth the cost.
Purpose
Without this, thread assignment fails the moment a character is referred to by pronoun. This is the hardest deterministic gap to bridge — this model fills it.
Where it fits
Runs after spaCy, enriches the character map. Output feeds into the thread assignment engine: each sentence's entity mentions are resolved to canonical names before assignment.
Alternatives
coreferee — spaCy-native, maintained successor to neuralcoref; easier integration, lower accuracy. neuralcoref — abandoned, still functional on older spaCy. AllenNLP coref — strong accuracy, heavy dependency footprint. OntoNotes-fine-tuned BERT — custom HuggingFace model, needs more setup.
resolved entity clusters → each sentence now carries a canonical character_set → sentences passed to sentence-transformers as plain strings
▼
Stage 03 · Embeddings
sentence-transformers
all-MiniLM-L6-v2 · SBERT
Encodes each sentence into a dense vector that captures semantic meaning. Sentences about the same topic or character cluster together in embedding space.
PyTorch · bi-encoder
Description
Siamese BERT network fine-tuned for semantic similarity. all-MiniLM-L6-v2 is 80MB, 6 layers, runs on CPU at useful speed. Produces 384-dim vectors. Cosine similarity between vectors = semantic closeness. This is the backbone of thread continuity detection.
Purpose
Two things: (1) measure semantic drift between sentences to detect thread shifts, (2) provide vectors for domain classification when passed to the zero-shot classifier.
Where it fits
Central hub. Embeddings flow both to the thread clustering stage and can be reused by the domain classifier as query vectors. One encode pass, two consumers.
Alternatives
all-mpnet-base-v2 — higher accuracy, slower, 420MB. paraphrase-MiniLM-L3-v2 — even smaller/faster, lower quality. OpenAI text-embedding-3-small — API call, excellent quality, not local. Cohere embed — API, strong multilingual. nomic-embed-text — strong local option, Apache 2.0.
384-dim float32 vectors → splits into two parallel paths: (A) cosine sim matrix → clustering, (B) vectors + raw text → zero-shot classifier
▼
Stage 04A · Classification
bart-large-mnli
facebook/bart-large-mnli
Zero-shot domain labelling. Classifies any paragraph against arbitrary labels — "technology", "finance", "romance" — without any training data.
PyTorch · NLI-based
Description
Natural Language Inference model repurposed as a classifier. For each candidate label, it scores the entailment probability of "This text is about [label]". No fine-tuning required. Returns a ranked list of labels with confidence scores. ~1.6GB model.
Purpose
Answers "is this paragraph in the tech domain?" with a confidence score. Can handle multi-label classification — a paragraph can score 0.82 on "technology" and 0.61 on "finance" simultaneously.
Where it fits
Parallel branch from the embedding stage. Receives the raw sentence text (not the vector — BART has its own encoder). Outputs domain labels that tag each sentence/paragraph before final aggregation.
Alternatives
SetFit — few-shot fine-tuning on sentence-transformers; 8–20 labelled examples, much faster inference, better accuracy once you have signal. deberta-v3-large-zeroshot — stronger NLI backbone, better zero-shot. GPT-4o via API — overkill for this task but near-perfect. KeyBERT — keyword extraction as a proxy for domain; deterministic-ish.
domain label + confidence per sentence → merged with character_set and embedding vector into a unified sentence record
▼
Stage 04B · Threading
Cosine Sim Clustering
scipy / numpy · custom
Groups sentences into coherent plotlines using semantic similarity + character co-occurrence. Produces a thread_id per sentence.
deterministic · no model
Description
Builds a sentence × sentence cosine similarity matrix from the SBERT embeddings. Combines this with the character_set from FastCoref — sentences sharing a character AND high semantic similarity are strong thread candidates. Agglomerative clustering groups them. Temporal/spatial shift markers (detected by regex in the deterministic pre-pass) act as hard thread-break signals.
Purpose
The actual plotline detector. Answers "which sentences belong to thread A, B, or C?" Each sentence receives a thread_id. This is the core deliverable for short-story analysis.
Where it fits
Parallel branch from embeddings, merges with co-reference output. Final consumer before aggregation.
Alternatives
HDBSCAN — density clustering, better with variable thread sizes, no need to specify N clusters. KMeans — requires knowing thread count upfront, fast. TextTiling — classic NLP algorithm for topic segmentation, fully deterministic, lower accuracy. BERTopic — topic modelling over embeddings, outputs topic labels rather than thread IDs.
thread_id + domain_label + character_set per sentence → merged into unified sentence records → passed to deterministic post-validator
▼
Stage 05 · Validation
Deterministic
Post-Validator
custom · rule-based
Sanity-checks model outputs against structural rules. Catches hallucinated threads, orphaned characters, and domain label inconsistencies.
deterministic · auditable
Description
Rule-based validation layer. Checks: (1) thread count is stable — no thread appears for one sentence then vanishes. (2) Characters don't teleport — same character in two threads simultaneously signals a co-reference error. (3) Domain labels are locally consistent — a single sentence labelled "tech" inside 8 "romance" sentences is likely a false positive. Applies majority-window smoothing.
Purpose
The "human at the edge" checkpoint. Flags anomalies as warnings rather than silently passing bad data downstream. Produces a clean, validated List[SentenceRecord].
Where it fits
Final stage before output. Enforces the invariants you'd check by hand. Outputs can be serialised to JSON for agent consumption or rendered as annotated text.
Alternatives
Pydantic validation — schema-level checks on the output structure. Great Expectations — data quality framework, overkill but powerful for production. Custom unit tests — pytest fixtures on known paragraph fixtures with expected thread/domain outputs.
List[SentenceRecord] → {text, thread_id, domain_labels, characters, confidence} → serialised output

Stage 06 · Output
Structured Output
JSON / annotated text
Per-sentence records with thread ID, domain label, resolved characters, and confidence scores. Machine-readable. Agent-ready.
deterministic
Schema
{
  "sentences": [
    {
      "text": "str",
      "thread_id": "int",
      "domain": "[{label, score}]",
      "characters": "[canonical_name]",
      "flags": "[warning_str]"
    }
  ]
}
Consumers
Downstream agents, summarizer pipeline (AGENT-SUM), retrieval systems, human review interfaces, evaluation harnesses.
